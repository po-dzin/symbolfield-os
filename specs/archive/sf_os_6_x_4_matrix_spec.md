# SF HUD — 6×4 Matrix Specification v0.1

> Цель: зафиксировать **один живой орган** системы — SF HUD — через матрицу 6×4 (6 слоёв × ○ ∣ • ∴), чтобы дальше можно было спокойно дорабатывать её и выгружать в Obsidian.

В этом документе — **только**:

- цель модуля SF HUD,
- матрица-спецификация 6×4,
- краткое описание функций HUD,
- стратегия поддержки и развития модуля.

---

## 1. Цель модуля SF HUD

SF HUD — это ежедневный рабочий интерфейс G, который:

- даёт **одну точку входа** в день/сессию,
- связывает действия (ритуалы, квесты, заметки) с **состоянием** и **XP**,
- позволяет видеть **паттерны** (что реально работает, что нет),
- остаётся достаточно простым, чтобы его хотелось открывать каждый день.

Этот документ — **дизайн-спецификация** HUD, а не лог.

---

## 2. Матрица 6×4: слои × ○ ∣ • ∴

Таблица ниже описывает, что делает HUD на каждом слое системы через четыре оси:

- `○` — форма (что хранится / структура),
- `∣` — поток (что делается / меняется),
- `•` — внимание (что осмысляется),
- `∴` — интеграл (какой вывод/результат собирается).

Эта матрица — **контракт** для дизайна HUD.&#x20;

| Слой / Ось            | `○` Форма (что хранится)                                                                                                                              | `∣` Поток (что делается)                                                                                                               | `•` Внимание (что осмысляется)                                                                                  | `∴` Интеграл (что собирается)                                                                                                 |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **1. Онтологический** | Список сущностей HUD: типы `Mode`, `Tone`, `Glyph`, `Ritual`, `Quest`, `Echo`, `XP`, `SemPhase`, теги SSA5 и т.п. Хранятся как справочники/константы. | Выбор активных сущностей в сессии: текущий `/mode`, активный `/quest`, текущая фаза SEM7, выбранный SSA5-тег для `/echo`.              | Вопрос: «в какой игре я сейчас?» — краткий контекст поля/дня (например: R&D, Center Way ops, rest-day).         | 1 строка онтологического итога: тег среды/мира сессии: `∴ world: ...` (для дальнейшего анализа паттернов).                    |

| **5. Психотелесный**  | Базовые показатели состояния G внутри HUD: `/tone`, опциональные поля «энергия/ясность», связь ритуалов с телом (•/∣/○).                              | Проживание через действия: выполнение ритуалов, закрытие micro-steps квеста, появление /echo → реальные микросдвиги в теле/уме.        | `Echo` как зеркало: 1–3 строки про телесно-психическое состояние (при желании с SSA5-тегом), без длинных эссе.  | Психо-итог: краткая строка `∴ state: ...` (что сдвинулось по телу/уму/ясности), пригодная для последующего просмотра сериями. |

Энергетический** | Метрики живости HUD: суммарный XP за сессию/день, количество ритуалов/шагов/echo, streaks использования (подряд дни/сессии).                          | Динамика использования: частота открытий HUD, плотность действий по dayline, распределение фокуса (больше ритуалов или больше квеста). | Вопрос: «живёт ли орган или просто существует как интерфейс?» — честная оценка энергетики взаимодействия с HUD. | Энергетические решения: настройки ритма (`утренний слот`, `1 глубокая сессия/день`, «без вечернего логирования» и т.п.).      |

## 3. Функции SF HUD (кратко)

1. **Оперативный интерфейс дня/сессии**

   - точка входа: выбор `/mode`, `/tone`, `/glyph`, активного `/quest`;
   - ядро: ритуалы (тело/дыхание/движение), micro-steps квеста, /echo;
   - выход: /close log + минимальный ∴-итог.

2. **Трекер состояний и XP**

   - связь ритуалов и действий с HP/CP/MP/DP;
   - минимальное отражение психотелесного состояния через /tone и /echo;
   - подсветка фаз SEM7 как эво-компаса.

3. **Поведенческий сенсор органа**

   - измерение живости: частота использования, глубина сессий, баланс между ритуалами и квестом;
   - возможность через ∴-строки видеть, как меняется стиль работы и жизни.

## 4. Стратегия поддержки и развития модуля SF HUD

1. **Этот документ = единственный контракт 6×4**

   - любые изменения логики HUD по слоям сначала фиксируются здесь (обновление формулировок в ячейках),
   - только потом переносятся в код / другие инструменты.

2. **Версионность**

   - при значимых изменениях матрицы 6×4 — повышать минорную версию (v0.1 → v0.2),
   - в Obsidian хранить копии по версиям: `SF HUD 6x4 v0.1`, `v0.2` и т.д.,
   - в коде HUD указывать, на какой версии матрицы он основан.

3. **Принцип развития**

   - не добавлять новые строки/столбцы без крайней необходимости;
   - работать сначала через **уточнение формулировок внутри ячеек** (сужение/прояснение);
   - изменения — только из практики (по следам реального использования HUD, а не из чистой теории).

4. **Связь с Obsidian**

- Obsidian-версия матрицы используется для более подробных примеров, списков и конкретных кейсов;
- корректно: раз в некоторое время синхронизировать канвас-версию и Obsidian-версию 6×4.

5. **Связь с кодом HUD**
   - разработка и рефакторинг HUD в коде должны опираться на текущую матрицу 6×4 как на источник правды;
   - любые поля/элементы интерфейса должны быть мапнуты хотя бы на одну ячейку матрицы.

Версия данного документа: **SF HUD 6×4 v0.1**.
